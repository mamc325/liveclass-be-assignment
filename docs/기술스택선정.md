# 라이브클래스 BE-A 과제 기술 선정 이유

> 작성일: 2026-05-20
>
> 과제: 라이브클래스 프로덕트 엔지니어 채용 과제 BE-A, 수강 신청 시스템
>
> 선정 원칙: 5일 내 완성 가능성, 동시성 제어 검증 가능성, 평가자 실행 환경 친화성

## 핵심 기술 판단

이번 과제의 핵심은 많은 기술을 도입하는 것이 아니라, 수강 신청 과정에서 정원 정합성을 깨뜨리지 않는 것이다.

따라서 기술 선택도 다음 기준에 맞춰 결정했다.

- 정원 초과 방지: PostgreSQL row lock + JPA `PESSIMISTIC_WRITE`
- 중복 신청 방지: 서비스 검증 + DB unique/partial unique index
- 스키마 재현성: Flyway migration
- 동시성 검증: Testcontainers 기반 PostgreSQL 통합 테스트
- 평가자 실행 편의성: Docker Compose + Swagger UI

---

## 1. Java 21 LTS

Java 21을 선택한다. Java 21은 Spring Boot 3.5.x에서 안정적으로 사용할 수 있는 LTS 버전이며, 기존 엔터프라이즈 Java/Spring 환경에서도 널리 사용되는 버전이다.

Java 25가 최신 LTS이지만, 채용 과제에서는 최신 버전 사용보다 평가자가 로컬에서 빠르게 실행할 수 있는지가 더 중요하다고 판단했다. 따라서 현재 프로젝트의 Gradle toolchain과 Spring Boot 3.5.x 호환성을 고려해 Java 21을 사용한다.

BE-A 과제의 핵심은 최신 문법 과시가 아니라 상태 전이, 정원 관리, 동시성 제어다. Java 21만으로도 record, enum, Stream API 등을 활용해 충분히 명확한 코드를 작성할 수 있다.

---

## 2. Gradle, Groovy DSL

현재 프로젝트가 Gradle Groovy DSL 기반으로 생성되어 있어 그대로 사용한다.

Maven은 보편적이지만 XML 설정이 길어질 수 있고, Gradle Kotlin DSL은 타입 안정성 장점이 있으나 과제 기간 내 트러블슈팅 비용을 줄이는 관점에서는 Groovy DSL이 더 실용적이라고 판단했다.

이번 과제는 빌드 설정이 복잡하지 않기 때문에 Groovy DSL의 동적 타이핑 단점은 제한적이다.

---

## 3. Spring Boot 3.5.x

현재 스캐폴드의 Spring Boot 버전은 3.5.x다.

Spring Boot 4.x가 이미 공개되어 있지만, 메이저 버전 변경은 Spring Framework 7 기반 전환과 관련 의존성 변화가 있어 5일 과제에는 불필요한 마이그레이션 리스크가 있다고 판단했다.

BE-A 과제에서 Spring Boot 4.x의 신규 기능이 핵심 평가 요소는 아니다. 따라서 안정성과 레퍼런스가 풍부한 Spring Boot 3.5.x를 유지한다.

README의 제약사항에는 "과제 제출 시점의 안정성과 현재 스캐폴드 호환성을 우선했다"고 명시할 예정이다.

---

## 4. PostgreSQL 16

동시성 제어가 핵심인 과제이므로 H2 대신 PostgreSQL을 사용한다.

H2는 가볍고 실행이 쉽지만, 실제 운영 DB와 락 동작, 트랜잭션 처리, 인덱스 기능이 다를 수 있다. 이번 과제는 "마지막 한 자리에 여러 사용자가 동시에 신청하는 상황"을 검증해야 하므로, 실제 DB 기준으로 동시성 테스트를 수행하는 것이 더 의미 있다고 판단했다.

PostgreSQL을 선택한 핵심 이유는 최신 DB 기능 사용이 아니라, 실제 DB의 행 락과 트랜잭션 격리 환경에서 수강 신청 정합성을 검증하기 위함이다.

정원 초과 방지는 인덱스만으로 해결하지 않는다. 수강 신청 시 `courses` 행을 `SELECT FOR UPDATE`로 잠근 뒤, 현재 정원 점유 수를 확인하고 신청 또는 대기열 등록을 하나의 트랜잭션에서 처리한다.

중복 신청 방지는 PostgreSQL의 unique index 또는 partial unique index로 보강할 수 있다.

```sql
CREATE UNIQUE INDEX uq_active_enrollment_per_user_course
ON enrollments(user_id, course_id)
WHERE status IN ('PENDING', 'CONFIRMED');
```

즉, 중복 신청은 DB 제약으로 보강하고, 정원 관리는 트랜잭션과 행 락으로 보장한다.

## 5. Spring Data JPA

BE-A는 복잡한 리포팅 쿼리보다 도메인 상태 전이가 중요한 과제다.

JPA를 사용하면 `course.open()`, `course.close()`, `enrollment.confirm()`, `enrollment.cancel()`처럼 상태 변경 규칙을 도메인 객체 내부에 캡슐화할 수 있다.

또한 `@Lock(LockModeType.PESSIMISTIC_WRITE)`를 통해 수강 신청 시 필요한 행 락을 선언적으로 표현할 수 있다.

이번 과제는 정산성 집계나 복잡한 SQL이 중심이 아니므로, MyBatis보다 JPA가 도메인 모델링과 구현 속도 측면에서 더 적합하다고 판단했다.

---

## 6. Flyway

`ddl-auto: update`는 빠르게 개발할 수 있지만, 스키마 변경 이력이 남지 않고 환경 간 DB 상태가 달라질 수 있다.

제출 과제라도 DB 스키마는 명시적으로 관리하는 편이 안전하다고 판단했다. 따라서 Flyway를 사용해 테이블, 제약조건, 인덱스, 목데이터를 migration 파일로 관리한다.

최종 설정은 다음 방향으로 가져간다.

```
spring.jpa.hibernate.ddl-auto=validate
spring.flyway.enabled=true
```

이렇게 하면 Flyway가 스키마를 생성하고, JPA는 Entity와 DB 스키마가 일치하는지만 검증한다.

또한 JPA 어노테이션만으로 표현하기 어려운 PostgreSQL partial index를 SQL로 직접 작성할 수 있다는 장점이 있다.

---

## 7. JUnit 5, AssertJ, Testcontainers

JUnit 5와 AssertJ는 Spring Boot 테스트의 기본 조합으로 사용한다.

Testcontainers는 실제 PostgreSQL 컨테이너를 띄워 통합 테스트를 수행하기 위해 사용한다. 특히 "마지막 한 자리에 여러 사용자가 동시에 신청하는 상황"은 H2보다 실제 PostgreSQL에서 검증하는 것이 더 의미 있다.

동시성 테스트는 `CountDownLatch`와 `ExecutorService`를 사용해 여러 요청을 동시에 시작시키고, 정원이 1명일 때 정확히 1명만 수강 신청에 성공하는지 검증한다.

대기열 기능을 포함하는 경우에는 다음 결과를 함께 검증한다.

- 정원 1명 강의에 10명이 동시에 신청
- Enrollment는 1건만 생성
- 나머지 사용자는 Waitlist에 등록
- Course의 enrolledCount는 1 유지
- 정원 초과가 발생하지 않음

## 8. Springdoc OpenAPI 2.8.x

BE 과제는 API 명세 또는 샘플 요청/응답 제출이 필요하다.

Springdoc OpenAPI를 사용하면 Swagger UI를 통해 API 목록, 요청 파라미터, 응답 구조를 빠르게 확인할 수 있다. 이는 평가자가 프로젝트를 실행한 뒤 API를 직접 테스트하는 데 도움을 준다.

Spring Boot 3.5.x와의 호환성을 고려해 Springdoc OpenAPI 2.8.x를 사용한다.

단, Swagger UI만 의존하지 않고 README에도 주요 API의 샘플 요청과 응답을 함께 작성한다.

---

## 9. Docker Compose

평가자가 로컬에 PostgreSQL을 직접 설치하지 않아도 실행할 수 있도록 Docker Compose를 사용한다.

PostgreSQL 이미지는 `postgres:16-alpine`처럼 버전을 고정한다. `latest` 태그는 평가 시점에 다른 버전이 내려올 수 있어 재현성이 떨어지므로 사용하지 않는다.

DB healthcheck를 추가해 PostgreSQL 준비 상태를 명확히 표현하고, README에는 다음 실행 순서를 안내한다.

```bash
docker compose up -d
./gradlew bootRun
```

또는 테스트 실행은 다음처럼 안내한다.

```bash
./gradlew test
```

## 도입하지 않는 기술

### Spring Security

과제에서 인증/인가는 간략히 처리해도 된다고 명시되어 있으므로 Spring Security는 도입하지 않는다.

대신 `X-USER-ID` 헤더로 사용자 ID를 전달받고, 서비스 계층에서 사용자 존재 여부, 역할, 본인 신청 여부, 강의 생성자 여부를 검증한다.

이번 과제의 핵심은 인증/인가 체계가 아니라 수강 신청 상태 전이와 정원 정합성이므로, Spring Security 도입은 과제 범위 대비 복잡도가 높다고 판단했다.

---

### Redis

정원 관리 동시성은 PostgreSQL의 행 락과 트랜잭션으로 충분히 해결 가능하다고 판단했다.

Redis 분산 락은 다중 애플리케이션 인스턴스나 더 높은 트래픽 상황에서는 고려할 수 있지만, 이번 과제 범위에서는 과한 복잡도다.

또한 DB에 최종적으로 저장되는 정원과 신청 데이터의 정합성이 핵심이므로, 같은 데이터베이스 트랜잭션 안에서 `Course` 행을 잠그고 신청/대기열 처리를 수행하는 편이 설명과 검증이 명확하다.

---

### H2

H2는 실행이 간단하지만 PostgreSQL의 partial index와 락 동작을 정확히 재현하기 어렵다.

이번 과제는 동시성 제어가 핵심이므로 테스트와 로컬 실행 모두 PostgreSQL 기준으로 맞춘다.

다만 평가자 실행 편의성을 위해 PostgreSQL은 Docker Compose로 제공한다.

---

## 요약

이번 기술 선택의 핵심은 "짧은 기간 안에 동작하는 시스템을 만들되, 과제의 핵심인 동시성 제어는 실제 DB 기준으로 검증한다"이다.

Java 21, Spring Boot 3.5.x, JPA는 구현 안정성과 생산성을 확보하기 위한 선택이다.

PostgreSQL, Flyway, Testcontainers는 정원 관리와 중복 신청 방지 같은 비즈니스 규칙을 더 신뢰성 있게 검증하기 위한 선택이다.

평가자 입장에서는 Docker Compose와 Swagger UI를 통해 프로젝트를 빠르게 실행하고 API를 확인할 수 있다.
