# 기능 명세: 수강 신청 시스템

**기능**: `001-enrollment-system`
**상태**: 진행 중
**작성일**: 2026-05-21
**주요 출처**: 기존 `docs/` 설계 문서

## 목표

크리에이터가 강의를 등록하고, 수강생이 강의 조회, 수강 신청, 대기 등록, 결제 확정, 취소를 수행할 수 있는 라이브클래스 수강 신청 API를 구현한다. 모든 흐름은 정원, 소유권, 상태 전이 규칙을 위반하지 않아야 한다.

## 사용자

- 크리에이터: 강의를 등록하고 모집을 시작/마감하며, 본인 강의의 수강생/대기자 목록을 조회한다.
- 수강생: 강의를 조회하고, 모집 중인 강의에 신청하거나 대기하며, 결제 확정/취소를 수행하고 본인 신청/대기 목록을 조회한다.

## 사용자 시나리오

### P1 - 수강생이 모집 중인 강의에 신청한다

잔여 정원이 있는 `OPEN` 강의에 수강생이 신청하면, 시스템은 `Enrollment(PENDING)`을 생성하고 `occupied_count`를 1 증가시킨 뒤 신청 성공 응답을 반환한다.

### P1 - 정원이 가득 찬 강의에 신청하면 대기자가 된다

잔여 정원이 없는 `OPEN` 강의에 수강생이 신청하면, 시스템은 `Waitlist(WAITING)`을 생성하고 현재 대기 순번을 포함한 응답을 반환한다. 정원 점유 수는 바뀌지 않는다.

### P1 - 동시 신청 상황에서도 정합성을 유지한다

여러 수강생이 마지막 잔여 자리에 동시에 신청해도, 활성 신청 수는 강의 정원을 초과하지 않아야 한다. 초과한 유효 요청은 대기열로 이동한다.

### P1 - 수강생이 결제를 확정한다

수강생 본인의 `PENDING` 신청에 대해 결제 확정을 요청하면, 신청 상태는 `CONFIRMED`가 된다. 강의 정원 점유 수는 바뀌지 않는다.

### P1 - 수강생이 수강 신청을 취소한다

수강생 본인의 `PENDING` 신청 또는 취소 가능한 `CONFIRMED` 신청을 취소하면, 신청 상태는 `CANCELLED`가 되고 정원이 복구된다. 강의가 아직 `OPEN`이면 가장 오래된 대기자가 자동 승격된다.

### P2 - 크리에이터가 강의 생명주기를 관리한다

크리에이터는 본인의 `DRAFT` 강의를 `OPEN`으로 전환할 수 있고, 본인의 `OPEN` 강의를 `CLOSED`로 전환할 수 있다. 마감된 강의는 신규 신청을 받지 않지만 기존 신청의 결제 확정과 유효한 취소는 허용한다.

### P2 - 역할과 소유권을 검증한다

수강생은 강의를 생성/관리할 수 없고, 크리에이터는 수강생 역할의 신청 API를 사용할 수 없다. 사용자는 타인의 리소스를 변경할 수 없다.

## 기능 요구사항

- FR-001: 시스템은 `X-USER-ID`로 호출자를 식별하고 DB에서 사용자의 역할을 조회해야 한다.
- FR-002: 크리에이터는 항상 `DRAFT` 상태의 강의만 생성할 수 있다.
- FR-003: 크리에이터는 본인의 강의를 `DRAFT`에서 `OPEN`으로 전환할 수 있다.
- FR-004: 크리에이터는 본인의 강의를 `OPEN`에서 `CLOSED`로 전환할 수 있다.
- FR-005: 수강생은 `OPEN` 강의에만 신청할 수 있다.
- FR-006: 같은 수강생은 같은 강의에 활성 `PENDING` 또는 `CONFIRMED` 신청을 2개 이상 가질 수 없다.
- FR-007: 같은 수강생은 같은 강의에 활성 `WAITING` 대기를 2개 이상 가질 수 없다.
- FR-008: 정원이 남아 있으면 신청 시 `Enrollment(PENDING)`을 생성하고 `Course.occupiedCount`를 증가시킨다.
- FR-009: 정원이 가득 찼으면 신청 시 `Waitlist(WAITING)`을 생성하고 `Course.occupiedCount`는 변경하지 않는다.
- FR-010: `PENDING` 신청은 소유자가 결제 확정할 수 있다.
- FR-011: `PENDING` 신청은 소유자가 기간 제한 없이 취소할 수 있다.
- FR-012: `CONFIRMED` 신청은 `confirmedAt + cancellationDeadlineDays` 이내에만 소유자가 취소할 수 있다.
- FR-013: 신청 취소는 `Course.occupiedCount`를 감소시킨다.
- FR-014: `OPEN` 강의의 신청 취소는 존재하는 가장 오래된 `WAITING` 대기자를 자동 승격한다.
- FR-015: 자동 승격은 대기 상태를 `PROMOTED`로 바꾸고 `promotedFromWaitlistId`를 가진 새 `Enrollment(PENDING)`을 생성한다.
- FR-016: 대기 취소는 `WAITING` 상태에서만 허용한다.
- FR-017: 강의, 신청, 대기 목록 API는 안정적인 페이지네이션 정렬을 제공해야 한다.
- FR-018: 모든 도메인/API 실패는 안정적인 `code`를 가진 ProblemDetail 형식으로 응답해야 한다.

## 핵심 엔티티

- `User`: `CREATOR` 또는 `STUDENT`; 과제용 시드 기반 인증 모델.
- `Course`: 강의 정보, 정원, 점유 수, 상태, 크리에이터 소유권, 취소 가능 기간.
- `Enrollment`: 활성 또는 이력 수강 신청 상태. 대기열 자동 승격으로 생성된 경우 대기 row와 연결된다.
- `Waitlist`: 대기 중, 승격됨, 취소됨 상태를 가진 대기열 항목.

## 확정된 결정

- 결제는 confirm API로 시뮬레이션하며 외부 PG 연동은 범위 밖이다.
- 정원은 결제 확정 시점이 아니라 수강 신청 생성 시점에 선점한다.
- 결제 실패는 현재 범위에서 별도 상태로 모델링하지 않는다.
- 취소 row는 활성 중복 제약에서 제외되므로 취소 후 재신청은 허용한다.
- 강의 마감은 신규 신청과 자동 승격을 막지만 기존 `PENDING`의 결제 확정과 유효한 취소는 막지 않는다.
- 단일 인스턴스 과제 범위이므로 Redis/분산 락은 제외한다.

## 범위 제외

- Spring Security 로그인/세션/토큰 흐름.
- 회원가입 또는 User CRUD.
- 외부 결제 PG 연동.
- 강의 수정/삭제 API.
- 분산 락 인프라.
- 다국어 메시지.

## 인수 기준

- AC-001: 크리에이터가 `POST /api/courses`를 호출하면 `DRAFT` 강의가 생성된다.
- AC-002: `POST /api/courses/{id}/open`은 본인 소유의 `DRAFT` 강의만 모집 시작한다.
- AC-003: `POST /api/courses/{id}/enrollments`는 정원이 남아 있으면 신청 성공을 반환한다.
- AC-004: 같은 엔드포인트는 정원이 가득 차 있으면 대기 등록을 반환한다.
- AC-005: 활성 신청/대기 중복 시도는 문서화된 409 코드로 실패한다.
- AC-006: `POST /api/enrollments/{id}/confirm`은 본인 소유의 `PENDING` 신청만 확정한다.
- AC-007: `POST /api/enrollments/{id}/cancel`은 정원을 복구하고, `OPEN` 강의라면 가장 오래된 대기자를 승격한다.
- AC-008: `POST /api/waitlists/{id}/cancel`은 본인 소유의 `WAITING` 대기만 취소한다.
- AC-009: 마지막 자리 동시 신청 테스트에서 활성 신청 수가 정원을 넘지 않음을 증명한다.
- AC-010: 에러 응답은 `status`, `title`, `detail`, `instance`, `code`를 포함한다.
